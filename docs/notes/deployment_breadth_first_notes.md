# Fight Detection 项目广度优先学习笔记

这份笔记适合当前时间紧、想先扩展技术广度的阶段。

你的目标暂时不是把模型做得更强，而是先学会下面两类能力：

1. 如何把一个 Python 视觉项目整理成“可部署”的结构。
2. 如何接触 C++ 推理、开发板部署、模型转换这些工程技术。

这条路线和 student_project_checklist.md 不冲突。
区别只是顺序调整为：

- 先学部署和工程化的大框架。
- 再回头研究数据、实验和模型细节。

## 一、先明确你当前项目离部署还差什么

当前项目已经有这些基础：

- 有训练脚本。
- 有导出的 MindIR 模型。
- 有实时推理 demo。
- 有 YOLO pose 作为前处理。

但当前项目还不具备真正的“部署形态”，因为它还存在这些特点：

- 目前主入口是 Python + Streamlit 页面，不是服务程序。
- 没有 C++ 代码。
- 没有 CMake 或编译工程。
- 没有模型转换脚本。
- 没有针对开发板的依赖裁剪。
- 前处理和后处理都直接写在 Python 逻辑里。

所以你接下来学的重点，不是直接把整个项目硬搬到板子上，而是先学会把项目拆成适合部署的结构。

## 二、你现在最该掌握的 4 个部署概念

### 1. 训练和推理必须分离

你要开始形成一个固定意识：

- 训练代码只在开发机运行。
- 部署端只保留推理代码。
- 开发板上通常不训练，只加载导出的模型。

对你这个项目来说：

- models/training/dataset_loader.py 属于训练侧。
- src/predictions.py 和 app.py 更接近推理侧。

### 2. 页面不是部署本体

现在 app.py 是演示界面，不是部署架构本身。

以后真正部署时，通常会变成：

- 视频输入模块。
- 前处理模块。
- 模型推理模块。
- 告警输出模块。

页面只是其中一个可选外壳。

### 3. 模型能导出，不等于能部署

你现在已经有 version3.mindir，这很好，但要明白：

- 导出模型只是第一步。
- 真正部署还要看目标设备是否支持这个格式。
- 还要看前处理和后处理怎么落地。

### 4. 板端最重的往往不是分类器，而是前处理

这个项目里，真正更可能成为瓶颈的是 YOLO pose，而不是后面的 GRU。

因为：

- GRU 输入只有 41 x 153，计算量不大。
- YOLO pose 需要逐帧做人体关键点检测，算力开销大很多。

所以以后迁移时，要优先关注 pose 检测能不能跑，而不是只关注分类模型。

## 三、广度优先时，你应该按什么顺序学

### 第一阶段：先学“部署结构”而不是先学板子

目标：知道一个视觉项目怎么从研究代码变成部署代码。

你先要会回答这些问题：

- 什么是训练脚本，什么是推理脚本？
- 什么是模型导出？
- 什么是前处理、后处理？
- 什么是推理引擎？
- 为什么部署程序通常不带训练逻辑？

对这个项目，你现在可以做的事：

- 把当前项目拆成训练、推理、UI 三部分。
- 在笔记里画出真正的部署结构图。

建议你先不改模型，只先理清结构。

### 第二阶段：先学 Python 推理脚本的“无界面版”

目标：让项目先脱离 Streamlit，变成纯推理程序。

这是你进入 C++ 和开发板前最关键的一步。

为什么？

- 因为 C++ 或开发板部署，本质上都不关心 Streamlit 页面。
- 它们关心的是输入、预处理、推理、输出这条主链路。

你要先学会把当前项目改成：

- 输入一个视频文件。
- 输出逐帧或逐段预测结果。
- 可以保存结果视频或打印日志。

如果这一步没做好，直接学 C++ 很容易变成只会编译 Hello World。

### 第三阶段：学习最小 C++ 推理工程

目标：不是马上重写全项目，而是先理解 C++ 部署项目长什么样。

你需要先学会这些基础：

- CMake 是什么。
- C++ 项目怎么链接 OpenCV。
- 模型推理库怎么加载模型。
- 输入图像如何转成模型需要的 tensor。
- 推理结果如何做后处理。

对你这个项目最合理的切入方式是：

- 先不要在 C++ 里实现完整 YOLO pose + GRU。
- 先拿“已经提取好的特征”或“简化输入”跑一个最小推理 demo。

也就是说，先学会：

- C++ 读取一个特征文件。
- C++ 调用推理运行时加载模型。
- C++ 输出一个分类结果。

先把链条走通，比一开始就追求完整系统更重要。

### 第四阶段：再学习开发板部署

目标：理解交叉编译、架构兼容、依赖裁剪和板端调试。

这一步你要重点理解的是这些关键词：

- ARM 架构。
- 交叉编译。
- 板端运行时环境。
- 模型转换。
- 板端摄像头采集。
- 性能 profiling。

你要先接受一个现实：

- 板端部署不是“把 Python 文件复制过去”。
- 板端部署本质上是重新设计运行环境。

### 路线 A：最快建立部署视野

这是我最推荐你的路线。

## 五、如果你要先学 C++，应该具体学什么

你不是为了学语法而学 C++，而是为了“视觉推理部署”而学 C++。

所以重点不是 STL 全家桶，而是下面这些：

### 必学 1：CMake 基础

至少要会：

- 创建一个最小 CMake 项目。
- 添加源文件。
- 链接 OpenCV。
- 生成可执行文件。

### 必学 2：OpenCV C++ 基础

至少要会：

- 读取图片和视频。
- 逐帧处理视频。
- resize、颜色空间转换、画框和写字。
- 保存处理后的视频。

### 必学 3：数组和 tensor 思维

至少要会：

- 图像在内存里是怎么排布的。
- HWC 和 CHW 的区别。
- float32 和 uint8 的区别。
- batch 维度是什么。

### 必学 4：部署程序结构

至少要会：

- 初始化模型。
- 读取输入。
- 预处理。
- 推理。
- 后处理。
- 输出结果。

你只要先掌握这 4 块，后面再接板子就会顺很多。

## 六、如果你要先学开发板部署，应该具体学什么

### 必学 1：设备架构意识

你要先知道：

- 开发机通常是 x86。
- 开发板通常是 ARM。
- 不是所有库都能直接复制过去运行。

### 必学 2：运行时意识

你要先知道：

- Python 能运行，不代表依赖能装齐。
- 模型格式能导出，不代表板端 runtime 支持。
- 有 NPU，不代表你当前模型就能直接上 NPU。

### 必学 3：性能意识

你要先知道：

- 视频任务看的是端到端速度。
- 真正要测的是 FPS、延迟、内存占用、温度。
- 板端部署往往需要降分辨率、降帧率、裁剪模型。

### 必学 4：最小验证意识

你以后上板时，应该按下面顺序验证：

1. 板子能不能读视频或摄像头。
2. 板子能不能加载模型。
3. 板子能不能跑单帧推理。
4. 板子能不能跑连续视频。
5. 板子能不能稳定跑一段时间。

不要一上来就追求完整系统联调。

## 七、你现在最适合做的“广度优先任务清单”

### 任务 1：学会把项目讲成部署结构

你要能把这个项目改写成下面这种表达：

- 输入端：摄像头或视频文件。
- 前处理端：YOLO pose 提取关键点。
- 序列建模端：41 帧滑动窗口。
- 推理端：MindIR 分类模型。
- 输出端：标签、概率、告警记录。

完成标准：

- 你可以从“部署视角”而不是“论文视角”讲项目。

### 任务 2：写一个无界面的推理脚本

目标：让项目不依赖 app.py 页面也能完成推理。

建议功能：

- 输入视频路径。
- 输出每段视频的预测结果。
- 可选保存结果视频。

完成标准：

- 你已经得到一个更接近部署程序的入口。

### 任务 3：单独分析哪些部分将来适合迁移到 C++

建议拆分：

- 视频读取：适合 C++。
- OpenCV 预处理：适合 C++。
- UI：不重要，可以先不管。
- 模型推理：需要看目标 runtime。
- YOLO pose：这是最大难点，要单独研究。

完成标准：

- 你知道不是所有模块都要同时迁移。

### 任务 4：搭一个最小 C++ + OpenCV 工程

这个任务和你当前项目不必强耦合，但非常值得先做。

建议目标：

- 读取一个视频。
- 逐帧显示或保存。
- 在画面上写字。

完成标准：

- 你知道 C++ 视觉程序如何被编译和运行。

### 任务 5：开始积累部署术语表

你应该把下面这些词都弄明白：

- 编译。
- 交叉编译。
- 动态库。
- 静态库。
- runtime。
- model export。
- pre-process。
- post-process。
- benchmark。

完成标准：

- 你后面看开发板教程时不会一头雾水。
