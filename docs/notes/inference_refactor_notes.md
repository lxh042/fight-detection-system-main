# 从修复 predictions.py 到推进 run_inference.py 的过程笔记

这份笔记记录了项目从“评估脚本存在运行问题”到“具备更接近部署形态的无界面推理入口”的完整过程。

它的目的不是单纯记录改了哪些代码，而是帮助我理解一条很重要的工程思路：

- 先把代码跑通。
- 再确认问题的真实原因。
- 最后把原型改造成更适合部署和迁移的结构。

---

## 一、起点：项目当时存在什么问题

在这个阶段，项目虽然已经具备训练、评估和实时页面三个部分，但真正跑起来时暴露了几个关键问题：

1. predictions.py 原本是按批量方式做推理评估。
2. 训练导出的 MindIR 模型虽然能加载，但实际推理输出并不符合 predictions.py 原本假设的批量行为。
3. app.py 虽然能代表完整推理链路，但它依赖 Streamlit 页面，不适合后续学习 C++、板端部署或服务化结构。

这意味着项目当时更像一个“能展示思路的原型”，而不是一个“适合继续迁移和部署的工程入口”。

---

## 二、第一步：先把训练和推理真的跑起来

为了避免“只看代码猜问题”，先用已有环境把训练和推理都实际跑了一遍。

### 1. 训练验证

使用用户指定的 conda 环境 pytorch 运行训练脚本：

- 训练入口：models/training/train_model.py

训练结果：

- 训练 100 个 epoch 正常完成。
- 成功导出了：
  - models/version3.ckpt
  - models/version3.mindir

最终验证指标为：

- acc = 0.7917
- recall = 0.7794
- f1 = 0.7891
- auc = 0.8735

混淆矩阵：

- [[164, 40],
-  [45, 159]]

### 2. 推理验证

接着运行离线评估脚本 predictions.py。

结果不是简单报环境错误，而是在评估阶段出现了样本数不一致的问题。这个问题非常关键，因为它说明：

- 脚本本身的推理逻辑假设，和实际导出的模型行为不一致。

这一步很重要，因为它不是“没装包”，而是真正的推理设计问题。

---

## 三、第二步：定位 predictions.py 为什么会失败

为了确认问题根因，没有直接拍脑袋改代码，而是先做了实际探测。

### 1. 原来的写法

predictions.py 原本按 batch 方式推理，大致思路是：

- 从测试特征里一次取一批样本。
- 把这一批样本一起送进 MindIR。
- 把输出拼接成所有预测结果。

这个写法本身是合理的，前提是模型真的支持批量输出。

### 2. 实际探测发现的问题

后来直接测试了不同 batch 输入下的模型输出形状，结果发现：

- 即使输入 batch 是 1、2、8、32、256，当前导出的 MindIR 仍然只返回形状为 (1, 1) 的输出。

这说明当前导出的模型图实际上是固定单样本行为。

也就是说，原脚本失败的根因不是 sklearn，不是 numpy，也不是环境问题，而是：

- predictions.py 假设模型支持批量评估。
- 但当前导出的 version3.mindir 实际只适合单样本推理。

这就是问题的根因。

---

## 四、第三步：把 predictions.py 改成可正常运行的单样本评估版本

定位完原因后，做了两项针对性修改。

### 修改 1：从批量推理改成单样本推理

修改文件：

- src/predictions.py

修改思路：

- 不再按 batch 把多个样本一起送入模型。
- 改成遍历测试集，每次只取一个样本。
- 每次输入形状固定为 1 x 41 x 153。
- 逐个收集输出概率，再计算整体指标。

这样做的好处是：

- 和当前导出的 MindIR 行为保持一致。
- 评估脚本不再因为输出长度不匹配而失败。

### 修改 2：优先复用测试特征缓存

进一步对 predictions.py 做了一个很实用的优化：

- 优先读取已有缓存：
  - preprocessed/test_features_v1.npz
  - 或 models/training/preprocessed/test_features_v1.npz
- 如果没有缓存，再回退到重新提取测试特征。

这样做的好处是：

- 每次评估不会重复跑一遍 YOLO 关键点提取。
- 评估速度更快，也更接近实际部署时的“模型评估脚本”定位。

### 修改后结果

修正后的 predictions.py 已经可以正常运行。

实际运行结果：

- acc = 0.7917
- recall = 0.7794
- f1 = 0.7891
- auc = 0.8735

混淆矩阵：

- [[164, 40],
-  [45, 159]]

这和训练结束时的指标一致，说明评估链路已经被真正修通。

---

## 五、第四步：为什么还要继续改 app.py 对应的推理链路

虽然 predictions.py 修好了，但这还只是“离线评估脚本可运行”。

后面如果要学：

- C++ 推理结构
- 开发板部署
- 模块化服务入口

那么 app.py 这种带页面的结构就不够用了。

原因很简单：

1. app.py 把页面逻辑和推理逻辑耦合在一起。
2. Streamlit 适合展示，不适合作为部署主入口。
3. 后面无论是 C++ 还是板端，真正需要的是“无界面推理程序”。

所以接下来的方向不是继续堆页面，而是把 app.py 的推理主链路抽出来。

---

## 六、第五步：从 app.py 抽出无界面推理入口 run_inference.py

### 目标

把原本在 app.py 里的核心推理流程，抽成一个不依赖 Streamlit 的命令行入口。

### 新增文件

- src/run_inference.py

### 这个文件保留了哪些核心逻辑

1. 输入源打开
   - 支持本地视频文件
   - 支持摄像头

2. 前处理
   - 每帧调用 YOLO pose 提取关键点
   - 最多保留 3 个人
   - 单帧特征长度 153

3. 时序建模
   - 维护 41 帧滑动窗口

4. 分类推理
   - 调用 version3.mindir
   - 计算 violence 概率
   - 使用滑动平均做平滑

5. 业务逻辑
   - 阈值判断
   - cooldown 去重
   - 记录事件触发

6. 输出
   - 命令行打印进度日志
   - 命令行打印最终摘要

### 这个文件去掉了什么

1. Streamlit 页面控件
2. 页面状态管理
3. HTML 告警框
4. 事件卡片组件
5. 图像在页面中的实时显示

这样做之后，这个文件就更像真正的部署入口了。

### 实际验证

用本地视频 fn.mp4 做短程测试，命令行运行成功。

得到的关键输出示例：

- [frame 1] collecting sequence buffer: 1/41
- [frame 20] collecting sequence buffer: 20/41
- [frame 40] collecting sequence buffer: 40/41
- [incident] #1 frame=41 ... prob=0.9664
- [frame 60] label=no-fight prob=0.0032 smoothed=0.1048 incidents=1

最终摘要：

- processed_frames = 60
- incident_count = 1
- elapsed_seconds = 4.09
- avg_fps = 14.69

这一步说明：

- 无界面推理入口已经跑通。
- 后续可以继续朝部署结构推进。

## 七、第六步：继续把 run_inference.py 往部署方向推进一层

如果只停留在“能命令行跑”，它仍然更像脚本，而不是部署程序。

所以后面又做了两类改造。

### 改造 1：关闭 YOLO 默认逐帧日志

修改文件：

- utills/frame_to_keypoits.py
- utills/data_preprocessing.py

修改内容：

- 调用 yoloModel.predict 时加上 verbose=False

这样做的原因是：

- YOLO 默认逐帧打印检测信息，会淹没真正有用的业务日志。
- 部署程序应该更关注自己的状态输出，而不是第三方库的逐帧噪声。

改完以后，命令行输出更清晰，更适合后面迁移和排障。

### 改造 2：把 run_inference.py 拆成更接近 C++ 的模块结构

新增文件：

- src/inference_core.py

现在结构变成：

- src/run_inference.py
  - 负责命令行参数入口
  - 负责调用核心流程

- src/inference_core.py
  - 负责模型加载
  - 负责输入源打开
  - 负责滑动窗口逻辑
  - 负责分类推理
  - 负责事件记录
  - 负责结果摘要

这一步很重要，因为它和后面做 C++ 时的思路是对应的。

后面如果真的做 C++，你大概率也会拆成：

- main.cpp 只负责入口和参数
- inference engine 负责推理主流程
- preprocessor 负责前处理
- output/logger 负责输出和日志

也就是说，这一步不是单纯“代码好看”，而是在训练部署结构意识。

## 八、第七步：给无界面入口补上可落盘产物

为了让它更接近真正的部署程序，又给无界面入口加了两个能力。

### 1. 保存结果视频

参数：

- --output-video

作用：

- 将带标签的推理结果保存成 mp4 视频。

意义：

- 这是部署后很常见的需求。
- 也方便后续做演示、复盘和错误分析。

### 2. 导出事件日志 JSON

参数：

- --event-log

作用：

- 将触发过的事件信息保存成 JSON。

记录内容包括：

- 事件序号
- 触发帧号
- 时间戳
- 原始概率
- 平滑概率
- 阈值

意义：

- 这非常接近真实系统的事件记录模块。

### 3. 导出汇总信息 JSON

参数：

- --summary-json

作用：

- 保存这次推理运行的整体摘要。

记录内容包括：

- 输入源
- 处理帧数
- 事件数
- 总耗时
- 平均 FPS
- 输出视频路径
- 事件日志路径
- 事件列表

意义：

- 这一步让脚本从“只会打印到终端”变成“能产出结构化结果”。
- 后续接 Web、接服务、接板端程序都会更方便。

## 十、这一段改造的真正意义是什么

如果只看表面，这段工作像是在做“脚本整理”。

但真正的意义其实是下面这几件事：

### 1. 学会先修真实问题，而不是空谈优化

predictions.py 的问题不是风格问题，而是模型导出行为和评估逻辑不一致。

这让我理解到：

- 工程里先要确认问题根因，再改代码。

### 2. 学会区分“展示入口”和“部署入口”

app.py 是展示入口。
run_inference.py 才更接近部署入口。

这让我理解到：

- 页面不是部署本体。
- 真正部署时要保留的是输入、前处理、推理、输出主链路。

### 3. 学会把原型改造成可迁移结构

引入 inference_core.py 以后，代码结构更像未来的 C++ 项目。

这让我理解到：

- 学部署不是先学板子，而是先学如何把代码拆成适合迁移的模块。

### 4. 学会让输出变成结构化结果

保存视频、事件 JSON、摘要 JSON 这几步非常重要。

因为这让我理解到：

- 真正的部署程序不能只靠终端日志。
- 它应该能留下可复盘、可对接、可分析的结果。

## 十三、如果继续往下走，最自然的下一步是什么

在这一阶段之后，后续最自然的方向有两个：

### 方向 1：继续往 C++ 思维靠拢

可以继续把当前 Python 结构进一步对齐成：

- 输入模块
- 前处理模块
- 推理模块
- 输出模块

这样后面做 C++ 对照会更直接。

### 方向 2：搭一个最小 C++ + OpenCV 工程

这时候已经很适合去做一个最小 C++ 工程了。

因为现在已经有了无界面 Python 版本作为对照样板，可以更清楚知道 C++ 程序应该模仿哪条链路。

## 十四、这一阶段我学到的核心一句话

从这个过程里，我学到最重要的一句话是：

- 部署不是把页面搬走，而是把推理主链路提炼出来，再把它整理成适合迁移和复用的结构。

这句话对后面继续学 C++、开发板、推理框架迁移都会很有帮助。
