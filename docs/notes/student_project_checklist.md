# Fight Detection 项目学习推进清单

这份笔记的目标不是把项目一下做大，而是让它持续训练你的计算机视觉能力。

当前最合适的方向：

- 先把现有项目彻底跑明白。
- 再做规范的实验和对比。
- 最后再考虑优化部署或迁移到开发板。

---

## 一、当前项目你已经具备的基础

这个项目已经包含了一个完整的小闭环：

- 数据集目录已经存在：data/train、data/test。
- 关键点提取已经存在：utills/frame_to_keypoits.py。
- 离线预处理已经存在：utills/data_preprocessing.py。
- 训练脚本已经存在：models/training/dataset_loader.py。
- 离线评估已经存在：src/predictions.py。
- 实时演示页面已经存在：app.py。

所以你现在不是“从零开始”，而是要进入“理解、整理、改进”的阶段。

---

## 二、总目标

通过这个项目，重点练 4 种能力：

1. 理解视频动作识别任务的完整流程。
2. 学会从数据和错误中分析模型问题。
3. 学会做最基本的模型对比实验。
4. 学会把一个 demo 整理成更规范的项目。

如果这 4 件事做到了，这个项目就已经很有学习价值。

---

## 三、第一阶段：先把当前项目整理到“可稳定使用”

这一阶段不要急着换模型，先修工程基本问题。

### 任务 1：修正硬编码路径

目标：让项目在当前仓库中直接运行，而不是依赖写死的本地路径。

要看和要改的文件：

- models/training/dataset_loader.py
- src/predictions.py
- app.py

检查清单：

- [ ] 去掉 dataset_loader.py 里写死的绝对路径。
- [ ] 统一模型目录名称，确认是 models 还是 Models，只保留一种。
- [ ] 所有模型路径都改成基于项目根目录的相对路径拼接。
- [ ] 检查 yolo11n-pose.pt 的加载路径是否稳定。

完成标准：

- 在当前仓库里，训练脚本和评估脚本不需要改路径就能运行。

### 任务 2：补全依赖

目标：让别人拿到项目后，按依赖文件就能安装环境。

要看和要改的文件：

- requirements.txt

检查清单：

- [ ] 确认 requirements.txt 是否包含 mindspore。
- [ ] 确认 requirements.txt 是否包含 tqdm。
- [ ] 检查 tensorflow 是否还真的需要。
- [ ] 给 readme 补一份更准确的依赖说明。

完成标准：

- requirements.txt 和代码实际依赖一致。

### 任务 3：把训练、评估、演示三条链路分清楚

目标：你能明确知道每个文件负责什么，不再混淆。

检查清单：

- [ ] 写清楚哪个文件负责训练。
- [ ] 写清楚哪个文件负责离线评估。
- [ ] 写清楚哪个文件负责实时演示。
- [ ] 写清楚数据从视频到特征再到分类结果的流向。

建议产出：

- 你自己画一张简单流程图，哪怕手画都可以。

完成标准：

- 你可以不用看代码，直接讲清整条 pipeline。

## 四、第二阶段：真正理解模型为什么对或为什么错

这一阶段开始训练你的分析能力。

### 任务 4：做错误分析

目标：不是只看 accuracy，而是找到模型真正容易错在哪。

检查清单：

- [ ] 找出一些被错分的 violence 视频。
- [ ] 找出一些被错分的 non-violence 视频。
- [ ] 观察这些视频里的人体关键点是否提取失败。
- [ ] 记录误报最常见的场景。
- [ ] 记录漏报最常见的场景。

建议你回答的问题：

- 是多人遮挡导致问题，还是动作不明显导致问题？
- 是视频质量差导致问题，还是序列长度不合适？
- 是 YOLO 关键点提取不稳，还是后面的 GRU 不够强？

建议产出：

- 一个 markdown 文件，名字可以叫 error_analysis.md。

完成标准：

- 你能列出当前项目最明显的 3 个失败原因。

### 任务 5：观察数据分布和特征形状

目标：真正理解模型输入长什么样。

要重点理解：

- 为什么单帧是 153 维。
- 为什么一个视频被固定成 41 帧。
- 为什么不足的人数和不足的帧数要补零。
- MinMaxScaler 对每个视频做归一化会带来什么影响。

检查清单：

- [ ] 打印一条样本的 shape。
- [ ] 随机看几条样本的数值范围。
- [ ] 检查补零帧占比高的视频是否更容易预测错。
- [ ] 看看 violence 和 non-violence 在特征上是否有明显差异。

完成标准：

- 你可以解释模型输入里的每一个维度代表什么。

## 五、第三阶段：做最基础的模型对比实验

这一阶段的目标不是追求最强模型，而是学会设计实验。

### 任务 6：建立 baseline

目标：把当前 GRU 结果固定成基线，后续所有改动都和它比。

检查清单：

- [ ] 记录当前模型的 acc、recall、f1、auc。
- [ ] 保存一份当前模型参数设置。
- [ ] 记录训练轮数、batch size、学习率。
- [ ] 固定随机种子，尽量减少实验波动。

建议产出：

- 一个实验记录表，可以先用 markdown 表格。

### 任务 7：只做一个模型对比

优先推荐下面 3 个里任选 1 个，不要同时做太多：

- [ ] GRU 对比 LSTM。
- [ ] 双层 GRU 对比单层 GRU。
- [ ] 序列长度 41 对比 21 或 61。

实验要求：

- 一次只改一个变量。
- 记录改动前后指标。
- 写出你对结果的解释。

完成标准：

- 你能说清楚这次实验改了什么、结果怎样、为什么会这样。

## 六、第四阶段：做一个“像研究”的特征改进

这是这个项目最值得你深入的地方，因为你做的是关键点时序，不只是图像分类。

### 任务 8：从原始关键点扩展特征

下面这些方向很适合学生练手，建议一次只选一个：

- [ ] 加入关键点速度特征。
- [ ] 把绝对坐标改成相对坐标。
- [ ] 只保留关键关节，而不是所有点。
- [ ] 尝试加入角度特征，比如手臂夹角、腿部夹角。

最推荐的起点：

- 先做“速度特征”。因为它最容易体现动作变化，对打斗这种任务也比较合理。

完成标准：

- 你能说明新特征是否提升了效果，以及为什么。

## 九、你每次改项目时都应该记录什么

以后每做一个改动，都写下下面 4 件事：

1. 我改了什么。
2. 我为什么要改。
3. 指标有没有变化。
4. 我认为变化的原因是什么。

如果你一直这样做，这个项目最后就不只是一个代码仓库，而是你自己的学习轨迹。

## 十一、最后提醒

你现在不需要追求“做出一个很厉害的大项目”，你更应该追求“让我通过这个项目变强”。

只要你最后能真正学会下面这些内容，这个项目就已经值了：

- 视频任务的数据处理流程。
- 关键点时序特征的设计思路。
- GRU/LSTM 这类时序模型的基本用法。
- 如何做实验对比和错误分析。
- 如何把一个原型项目整理得更规范。

这比单纯把页面做漂亮或者盲目换模型更重要。
